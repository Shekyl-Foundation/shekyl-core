Phase 2c reference vector — t8_vm_compute_hash_nop.bin
=======================================================

Spec attestation
----------------
T8 attests Rust `compute_hash` byte-for-byte parity with the v2
RandomX fork's `randomx_calculate_hash` end-to-end under stub-NOP
dispatch. Mirrors the production path exactly except the VM is
the Phase 2c `StubNopInterpretedLightVm` (per §5.6); both the
generator and the Rust port skip per-opcode bytecode dispatch but
run the full per-iteration loop body (spAddr derivation, register
loads, AES f/e mix, dataset reads, scratchpad writes, register
write-back), and chain `RANDOMX_PROGRAM_COUNT` (= 8) sub-programs
with Blake2b-256-over-RegisterFile re-seeding between programs and
a final `getFinalResult` AES hash + Blake2b finalization.

A T8 divergence isolates an integration bug at the
program-chaining / final-hash boundary that the per-stage tests
(T1–T7) couldn't catch — e.g. wrong RegisterFile layout passed to
the chaining Blake2b, wrong final hash size, wrong AES rounds
count in getFinalResult, wrong rounding-mode initialization. T8
is the single end-to-end attestation: if T1–T7 pass and T8 fails,
the bug is in the orchestration (`compute_hash` outer loop), not
in any individual stage.

Per RANDOMX_V2_PHASE2C_PLAN.md §5.7 (F7 T8) + §5.6 (stub-NOP
dispatch rationale) + §5.5 (compute_hash structure).

Output layout
-------------
32-byte Blake2b-256 final hash from `getFinalResult` (which runs
hashAes1Rx4 over the scratchpad → 64-byte hash buffer → Blake2b-256
over (RegisterFile || hash_buffer) → 32-byte final).

Total bytes for this vector: 32.

Notes — chain-boundary register-reset semantics
-----------------------------------------------
The C reference's `InterpretedVm::execute()` constructs a fresh
`NativeRegisterFile nreg;` per chain at `vm_interpreted.cpp:59`;
the `NativeRegisterFile` struct at `bytecode_machine.hpp:40`
declares `int_reg_t r[RegistersCount] = { 0 };`, so each chain's
iteration loop starts with the integer registers ZEROED regardless
of the prior chain's final `nreg.r` values. The end-of-chain
writeback at `vm_interpreted.cpp:130-131` stores `nreg.r` into
`reg.r` for the inter-chain Blake2b-256 input only — `reg.r` is
NOT loaded back into the next chain's `nreg.r`. The Rust port
fuses `reg` and `nreg` into a single `VmState.r` field per
`30-cryptography.mdc`'s secret-locality discipline; the per-chain
implicit reset that fell out of the C two-struct shape is
re-asserted as an explicit `self.r = [0; 8];` at the top of
`VmState::execute_program` (Rust port). T6/T7 single-chain spec-
vector tests don't surface this divergence (`VmState::new` returns
`self.r = [0; 8]` already); T8 is the first multi-chain test and
the one that pinned the discipline. See
`docs/design/RANDOMX_V2_PHASE2C_PLAN.md` §14 Round 0 R0-D10.
