Phase 2c reference vector — t6_vm_spaddr_4iter.bin
===================================================

Spec attestation
----------------
T6 attests Rust per-iteration `spAddr0` / `spAddr1` derivation
byte-for-byte parity with the v2 RandomX fork's interpreted-loop
spAddr computation under stub-NOP dispatch over the first 4
iterations of the 2048-iteration program loop.

Per-iteration spAddr derivation (vm_interpreted.cpp lines 70-74):

    spMix   = nreg.r[config.readReg0] ^ nreg.r[config.readReg1]
    spAddr0 ^= spMix
    spAddr0 &= ScratchpadL3Mask64       // (2 MiB - 64) bitmask
    spAddr1 ^= spMix >> 32
    spAddr1 &= ScratchpadL3Mask64

The snapshot fires BEFORE the iteration's register-load and AES
mix — capturing the spAddr values that the iteration uses to
index the scratchpad. A T6 divergence isolates the `readReg0/1`
indexing, the scratchpad-mask alignment, and the spMix XOR
sequence from the AES-mix substrate (T7).

The 4-iteration window is chosen because the spAddr derivation
reads `nreg.r[readReg0/1]` mutated by the previous iteration's
work, so the per-iteration values diverge after iteration 1
(iteration 0 uses the post-`initialize` zero registers; iteration
1 uses the iteration-0 dataset-read writeback; etc.). 4 iterations
covers (initial zero-state, post-AES-mix, post-second-AES-mix,
post-third-AES-mix) — enough to detect any per-iteration drift
that would compound over the full 2048.

Per RANDOMX_V2_PHASE2C_PLAN.md §5.7 (F7 T6) + §5.6 (stub-NOP
dispatch rationale).
